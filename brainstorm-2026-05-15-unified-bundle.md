# Brainstorm: Unified Single-File Bundle for @oda-components/components

**Date:** 2026-05-15  
**Status:** Complete  
**Scope:** Build configuration for packages/components to produce a single unified bundle

---

## 1. Current State Analysis

### Project Structure
- **Monorepo** managed by pnpm workspaces (pnpm 9.6.0)
- 10 internal packages under packages/: components, core, icons, styled, style-tokens, theme-tokens, themes, forms, metadata, auto-import-resolver
- The main library is packages/components (package name oda-components) containing **149 component directories** plus passthrough/ and umd/

### Current Build System
- **Rollup** with rollup-plugin-vue (v6 beta), @rollup/plugin-babel, postcss, terser
- Produces **one .mjs per component** (e.g., dist/button/index.mjs, dist/button/style/index.mjs)
- Also builds one ESM barrel (dist/index.mjs) and one UMD bundle (dist/umd/OdaComponents.min.js)
- Each component is a **separate rollup entry** via addFile() and addStyle() in rollup.config.mjs
- Total: approximately 300+ separate output files

### Current Barrel File (src/index.js)
- Auto-generated by scripts/prebuild.mjs (814 lines)
- Re-exports every component and style module via oda-components/<name> aliases
- Uses tsconfig.json path mapping: oda-components/* -> ./src/*
- The UMD build (src/umd/OdaComponents.js) uses relative imports and inlines all workspace deps

### Internal Dependencies
The components package depends on:
- @oda-components/core (BaseComponent, BaseDirective, config, utils)
- @oda-components/icons (~40 SVG icon .vue components)
- @oda-components/styled (styled runtime)
- @oda-components/utils (utility functions like cn)
- @oda-components/style-tokens (token-based CSS styles per component)

### Key Insight
The UMD build (addLibrary()) already does almost exactly what is requested: it takes a single entry, resolves all @oda-components/* aliases, and produces a single file with inlineDynamicImports: true. The ESM build, however, fragments into per-component files.

---

## 2. Evaluated Approaches

### Approach A: Vite Library Mode (Recommended)

Replace the Rollup config with vite.config.lib.ts using Vite built-in library mode.

**Pros:**
- Vite 8 is already a devDependency in the workspace
- @vitejs/plugin-vue v6 is already installed (replaces the outdated rollup-plugin-vue v6 beta)
- Simpler configuration -- one build.lib object rather than manually pushing entries
- Built-in CSS handling (no need for separate rollup-plugin-postcss)
- Native source map support
- Faster builds (Rolldown-powered in Vite 8)
- vite-plugin-dts can generate .d.ts rollup declarations automatically

**Cons:**
- Requires migrating away from the current rollup.config.mjs (moderate effort)
- Need to handle the alias resolution that the current config does

**Bundle output:** Single oda-components.mjs (ESM) + oda-components.umd.js (UMD) + optional style.css + index.d.ts (types)

### Approach B: Keep Rollup, Modify for Single Entry

Keep the existing rollup.config.mjs but strip out addFile() and addStyle(), keeping only addLibrary(). Modify the ESM entry to inline all dependencies.

**Pros:**
- Minimal change -- the UMD path already works this way
- No new tooling to learn
- Lower risk of regressions

**Cons:**
- rollup-plugin-vue v6 beta is outdated and may break with Vue 3.5+
- More verbose configuration than Vite lib mode
- No built-in TypeScript declaration generation

### Approach C: tsup

Use tsup (already used for metadata package) with a Vue plugin.

**Cons:** tsup does not natively handle .vue SFCs. Not viable for 149 component library.

---

## 3. Recommended Solution: Approach A -- Vite Library Mode

### Rationale
Vite 8 is already installed, @vitejs/plugin-vue v6 is already present, and Vite library mode is purpose-built for exactly this scenario. It replaces the fragile rollup-plugin-vue beta, simplifies config, and produces cleaner output. Migration risk is low because the existing barrel file (src/index.js) already aggregates all exports.
---

## 4. Implementation Design

### 4.1 Entry Point Strategy

**Option 1 (Preferred): Rewrite barrel to use relative paths**

Change the prebuild script to generate:

```js
// Button
export * from "./button/Button.vue";
export { default as Button } from "./button/Button.vue";
export * from "./button/style/ButtonStyle.js";
export { default as ButtonStyle } from "./button/style/ButtonStyle.js";
```

This eliminates the need for the oda-components/* path alias at build time entirely.

**Option 2: Configure Vite resolve.alias for the barrel aliases**

Keep current barrel format, add alias in Vite config. Less recommended -- adds unnecessary indirection.

```json
    "quill": { "optional": true }
  }

```

**Key changes from current package.json:**
- Package name changed to @oda-components/components for the scoped import pattern
- exports field simplified to a single "." entry -- no per-component sub-paths
- main points to UMD, module points to ESM
- types points to the rolled-up declaration file
- Workspace dependencies (@oda-components/core, icons, styled, utils, style-tokens) are REMOVED since they get inlined into the bundle
- vue moves to peerDependencies; chart.js and quill become optional peer deps
- files restricts npm publish to dist/ only
- The publishConfig section is no longer needed

### 4.4 TypeScript Declarations Strategy

**Problem:** The project uses .js and .vue source files (Options API), not .ts. TypeScript declarations exist as hand-written .d.ts files alongside each component (e.g., Button.d.ts).

**Strategy with vite-plugin-dts:**
1. Install vite-plugin-dts
2. Use rollupTypes: true to merge all .d.ts files into a single dist/index.d.ts
3. Create a tsconfig.build.json with correct path mappings

```json
{
  "compilerOptions": {
    "target": "ESNext",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "declaration": true,
    "declarationDir": "./dist/types",
    "emitDeclarationOnly": true,
    "baseUrl": ".",
    "paths": {
      "oda-components/*": ["./src/*"],
      "@oda-components/core/*": ["../core/src/*"],
      "@oda-components/core": ["../core/src/index.d.ts"],
      "@oda-components/icons/*": ["../icons/src/*"]
    },
    "skipLibCheck": true
  },
  "include": ["src/**/*.d.ts", "src/**/*.vue"],
  "exclude": ["node_modules", "dist"]
}
```

**Pragmatic alternative (recommended as starting approach):** Keep the existing auto-generated src/index.d.ts barrel and simply copy it to dist/index.d.ts in a postbuild step. This works because the .d.ts files are hand-maintained, not compiler-generated.

### 4.5 CSS / Style Handling

The project does NOT use traditional CSS files. Styles are defined programmatically in *Style.js files using the PassThrough (PT) system and @oda-components/style-tokens. These are JavaScript class-name maps, not CSS strings.

**Strategy:** Set cssCodeSplit: false. Any SFC style block content gets collected into one file. The @oda-components/style-tokens JS values get inlined into the bundle.

### 4.6 Prebuild Script Modifications

The prebuild.mjs should be modified to generate the barrel with relative paths instead of alias paths:

**Current pattern:**
```js
export * from "oda-components/button";
export { default as Button } from "oda-components/button";
```

**New pattern:**
```js
export * from "./button/Button.vue";
export { default as Button } from "./button/Button.vue";
```

The same change applies to style imports. This is a simple modification in the prebuild script string templates.

## 5. Consumer Usage

After the unified build, consumers import as:

```ts
// Named imports -- all from a single entry
import { Button, DataTable, Dialog, Select, Toast } from "@oda-components/components";

// Style objects if needed
import { ButtonStyle, DataTableStyle } from "@oda-components/components";

// Services and composables
import { useConfirm, useToast, useDialog } from "@oda-components/components";
```

**Naming note:** The current barrel exports components by their original names (Button, Accordion, DataTable). If OdaButton style naming is desired, the prebuild script should add prefixed exports:

```js
// Add both original and prefixed exports
export { default as OdaButton } from "./button/Button.vue";
export { default as Button } from "./button/Button.vue";
```

## 6. Potential Pitfalls and Mitigations

### 6.1 Bundle Size
- **Risk:** Inlining all 149 components + core + icons + styled into one file = estimated 1-3 MB minified, 300-800 KB gzipped.
- **Mitigation:** Acceptable trade-off. sideEffects: false enables tree-shaking in consuming bundlers. Unused components are stripped at consumer build time.

### 6.2 Circular Dependencies
- **Risk:** Current config silently suppresses circular dependency warnings.
- **Mitigation:** Audit with madge --circular. UMD build already inlines everything without issues, so this is likely under control.

### 6.3 chart.js / quill Externals
- **Risk:** Editor/Chart components fail at runtime if optional deps not installed.
- **Mitigation:** peerDependenciesMeta with optional: true. Document in README.

### 6.4 rollup-plugin-vue vs @vitejs/plugin-vue
- **Risk:** SFC compilation differences when switching from abandoned beta to maintained plugin.
- **Mitigation:** This is actually a benefit. @vitejs/plugin-vue is actively maintained and supports Vue 3.5+. Run test suite after migration.

### 6.5 Alias Resolution Completeness
- **Risk:** Missing an alias for an @oda-components/* import causes build failure.
- **Mitigation:** Alias list derived from existing ALIAS_ENTRIES in rollup.config.mjs. After barrel uses relative paths, only aliases used by component source files need resolution.

### 6.6 TypeScript Path Resolution in .d.ts
- **Risk:** Hand-written .d.ts files reference @oda-components/core types.
- **Mitigation:** Use vite-plugin-dts with rollupTypes, or pragmatically copy .d.ts files to dist/.

### 6.7 sideEffects and Vue Components
- **Risk:** sideEffects: false may cause tree-shaking to remove needed code.
- **Mitigation:** Components are explicitly imported by consumers. sideEffects: false is safe for the unified bundle.
